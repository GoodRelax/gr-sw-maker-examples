## Framework Improvement Analysis: SDD & Agentic SDLC Perspectives

After studying the full framework (process-rules, document-rules, spec-template, agent-list, review-standards, glossary), here are the key gaps organized by priority.

---

### A. SDD (Software Design Document) Gaps

**1. No formal SDD structure (High)**
The framework merges requirements and design into a single ANMS/ANPS spec. There's no IEEE 1016-style SDD that consolidates architecture decisions, component contracts, and interface specs into one authoritative design document. Design knowledge ends up scattered across `spec-architecture`, `openapi.yaml`, `security-architecture`, `observability-design`, and ADRs.

**Fix:** Create a lightweight SDD template that serves as a *design index* — not duplicating content, but providing a single entry point that links spec-architecture Ch3, API specs, security design, and ADRs with explicit requirement traceability.

**2. Design validation has no explicit gate (High)**
R1 reviews requirements quality. R2-R5 review implementation. But there's no formal review that verifies *"does the design cover all requirements?"* The traceability matrix is created but never reviewed as a quality gate.

**Fix:** Add an R-perspective (or extend R2) for "Design Completeness Review" — verify every FR/NFR traces to at least one design element before implementation begins.

**3. ADR traceability is one-directional (Medium)**
ADRs follow Nygard format but don't systematically link back to originating requirements or risks. When a requirement changes, there's no way to find affected ADRs without reading them all.

**Fix:** Extend ADR Form Block with `origin_requirement_ids` and `affected_components` fields.

**4. Component design contracts are undefined (Medium)**
Ch3.2 shows component boxes and dependency arrows, but no standardized template for *what each component promises* — its interface, error contract, and state behavior.

**Fix:** For ANPS-scale projects, create a component design template (responsibility, interface, dependencies, error contract, tested-by).

---

### B. Agentic SDLC Gaps

**5. No agent communication protocol (High)**
Agents communicate via file drops (handoff documents), but there's no protocol for how agents *request* information, signal blockage, or negotiate disagreements. This makes the framework implicitly sequential.

**Fix:** Define a lightweight protocol: requests go to `project-management/requests/`, responses update the same file, orchestrator monitors for timeouts.

**6. Agent conflict resolution is undefined (High)**
What happens when `review-agent` says "this architecture has a concurrency flaw" and `architect` disagrees? Currently: deadlock or implicit orchestrator override with no documented reasoning.

**Fix:** Add explicit escalation rules to process-rules Ch3: agents document positions in a shared decision record, orchestrator resolves using requirements + design principles + risk as criteria, losing position becomes "Alternative Considered" in the ADR.

**7. Context loss at agent handoffs (Medium)**
The handoff Form Block is minimal (from/to/status/phase). It doesn't carry *what the previous agent learned* — trade-offs evaluated, assumptions made, questions unanswered.

**Fix:** Extend handoff Form Block with `context_summary`, `assumptions`, `open_questions`, and `blocked_items`. This prevents the receiving agent from re-discovering what the previous agent already figured out.

**8. No agent performance metrics (Medium)**
Quality targets are project-level. There's no per-agent tracking — you can't tell if `architect` consistently produces designs with concurrency issues, or if `implementer` has a declining defect rate over phases.

**Fix:** Have `progress-monitor` track per-agent metrics (findings by severity, rework cycles, artifacts produced). Feed these into retrospectives for targeted prompt improvements.

**9. Agent scheduling/priority undefined (Medium)**
With 21+ agents and limited concurrency, which agents run first? The Phase Activation Map shows *which* agents are active, but not *in what order* or *which can truly parallelize*.

**Fix:** Create an agent-capability-matrix showing: minimum model requirement, parallelizable-with, depends-on. This drives WBS optimization and makes scheduling decisions transparent.

**10. No feedback loop from implementation to design (Medium)**
If `implementer` discovers a design flaw during coding, the only option is a generic defect ticket. There's no structured path back to `architect` to update the design.

**Fix:** Create a "design-issue-report" file type with fields for related design element, implementer assessment, and architect response. Route these to architect before design review gate.

---

### C. Priority Summary

| Priority | Gap | Category | Impact |
|----------|-----|----------|--------|
| P0 | No formal SDD structure / design index | SDD | Design knowledge fragmented |
| P0 | No agent communication protocol | Agentic | Agents can't negotiate, only pass files |
| P1 | No design completeness review gate | SDD | Design may miss requirements |
| P1 | Agent conflict resolution undefined | Agentic | Potential deadlocks |
| P1 | Context loss at handoffs | Agentic | Rework from re-discovery |
| P1 | Agent scheduling/priority undefined | Agentic | Unpredictable execution order |
| P2 | ADR traceability one-directional | SDD | Can't find affected decisions on change |
| P2 | Component design contracts missing | SDD | Implementers infer from folder structure |
| P2 | No agent performance metrics | Agentic | Can't improve specific agents |
| P2 | No implementation-to-design feedback | Agentic | Design assumptions accumulate |

### Key Takeaway

The framework is **strong on specification and requirements** (ANMS/ANPS is well-designed) but has two systemic gaps:

1. **Design is treated as part of the spec, not as a standalone discipline.** This works for small ANMS projects but breaks down at ANPS/ANGS scale where design decisions need independent review, traceability, and lifecycle management.

2. **Agents are defined as roles, not as collaborators.** The framework specifies *what* each agent owns but not *how* agents interact — no communication protocol, no conflict resolution, no context preservation, no performance feedback. This makes the agentic SDLC feel like "sequential automation" rather than "collaborative intelligence."

## Output Format

- **Enclose the entire Markdown with six backticks ```` `````` ```` at the beginning and end.** Specify its language as markdown.

- **Never output speculation or fabrications.** If something is unclear or requires investigation, explicitly state so.
- This method is called **MCBSMD** (Multiple Code Blocks in a Single Markdown)

### Code and Diagram Block Rules

- As a rule, use Mermaid for diagrams. Use PlantUML only when the diagram cannot be expressed in Mermaid.
- Any diagrams or software code inside the Markdown must each be enclosed in their own code blocks using triple backticks ` ``` `.
- Each code block must specify a language or file type (e.g., ` ```python ` or ` ```mermaid `).
- Each code or diagram block must be preceded by a descriptive title in the format **title:**
  (e.g., `**System Architecture:**`, `**Login Flow:**`)
- Always follow the structure below for every code or diagram block:

  > **title:**
  >
  > ```language
  > (code or diagram content here without truncation or abbreviation)
  > ```
  >
  > Write the explanation for the code block here, immediately after the block, following a blank line.

- Do not write explanations inside the code blocks.
- In all diagrams, prefer alphanumeric characters and underscores (`_`); non-ASCII text (no spaces) is allowed only when non-English is more appropriate for the diagram.
  Special symbols (e.g., `\`, `/`, `|`, `<`, `>`, `{`, `}`) are strictly prohibited.
- Output all diagram content without omission. Never use `...` or any shorthand.

### Diagram Label and Notation Rules

- All arrows and relationship lines in diagrams MUST have labels. Follow these notation rules:
  1. Mermaid `flowchart` and `graph`: place the label inside the arrow using pipes (e.g., `A -->|Label| B`)
  2. Other Mermaid diagrams / All PlantUML: place the label after the arrow using a colon (e.g., `A --> B : Label`)
- For line breaks in labels or node text:
  1. Mermaid: use `<br/>` inside a quoted string (e.g., `A -->|"Line1<br/>Line2"| B`, `A["Line1<br/>Line2"]`)
  2. PlantUML: use `\n` (e.g., `A -> B : Line1\nLine2`)

### Math Rules

- Use standard LaTeX notation for all mathematical formulas.
  1. Inline math: always use single dollar signs. Place a space before the opening `$` and a space after the closing `$` (e.g., `The function is $y = x + 1$ here.`)
  2. Block equations: always place `$$` on its own line, above and below the formula.
     Example:

     $$
     E = mc^2
     $$
