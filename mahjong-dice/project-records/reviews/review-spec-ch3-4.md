## 指摘一覧

### Medium 指摘

#### M-001: DRY違反の同期管理手順が未定義

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R2 (DRY) |
| 箇所 | ADR-002 (421-429行目)、3.3 File Structure |
| 問題 | index.html と dice-logic.js のコード重複は ADR-002 で意図的に許容されているが、「コード変更時に両方を同期する必要がある」ことへの具体的な管理手順（例: チェックリスト、自動比較スクリプト、CIでの差分検出）が未定義。Ch6 に「確認観点」の記載はあるが、実装フェーズで運用する具体的な手順が欠けている |
| 影響 | 実装フェーズでコード変更時に片方の同期を忘れ、本番 (index.html) とテスト対象 (dice-logic.js) の不一致が発生するリスクがある |
| 修正案 | ADR-002 の Consequences に具体的な同期手順を追記する。例: 「dice-logic.js を正とし、index.html への反映はビルドスクリプトまたはコピーチェックリストで管理する」 |

#### M-002: settled 状態のボタン遷移が曖昧

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R4 (状態遷移 - 中間状態の明示) |
| 箇所 | 3.4 状態遷移図 (312-323行目)、4.3 State Management (665-669行目) |
| 問題 | settled 状態のボタンが「disabled -> enabled へ遷移」と記載されているが、この遷移のタイミングが曖昧。状態遷移図では settled -> idle の遷移条件が「合計表示完了」だが、4.3 の表では settled 状態中にボタンが enabled になると読める。ボタン有効化が settled 状態中に起きるのか、idle 遷移時に起きるのか、設計が一意に解釈できない |
| 影響 | 実装者の解釈によって、ボタン有効化のタイミングが異なる実装になる可能性がある。シーケンス図 (370-372行目) では enableButton() の後に settled -> idle 遷移が発生しており、ボタン有効化は settled 状態内で行われる設計と読めるが、状態管理表との整合性が不明確 |
| 修正案 | 4.3 の状態管理表で settled 状態のボタン列を「disabled（ズームイン表示中） -> 表示完了後に idle へ遷移し enabled」と明確化する。または settled 状態終了時のコールバック内で enableButton() と状態遷移を同時に行う旨を明記する |

#### M-003: アニメーション中の非同期競合に対する設計方針が不足

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R4 (レースコンディション - 非同期処理間の競合) |
| 箇所 | 3.5 シーケンス図 (335-373行目)、4.3 State Management (671-674行目) |
| 問題 | requestAnimationFrame ベースのアニメーション中に以下の非同期イベントが発生した場合の設計方針が未記載: (1) ブラウザタブが非アクティブになった場合（rAF が停止/低速化する）、(2) ページの visibility 変更。disabled 属性による二重防御は記載されているが、アニメーションコールバック自体の中断・再開方針が不明 |
| 影響 | タブ切り替え時にアニメーションが不完全な状態で停止し、settled への遷移が発生しない可能性がある。結果としてボタンが永続的に disabled になるデッドロック類似の状態が生じうる |
| 修正案 | 以下のいずれかの方針を設計に追記する: (a) visibilitychange イベントでアニメーションを即座に完了させ settled へ遷移する、(b) アニメーションにタイムアウト上限を設け、超過時は強制完了する、(c) Limitations に「タブ非アクティブ時のアニメーション動作は保証しない」と明記する |

## 良好な点

1. **CA レイヤー仕訳の的確さ**: プロジェクト規模に対して適切な粒度（4層・7コンポーネント）で設計されている。過度な抽象化を避け、関数ベースの分離にとどめた判断は KISS に合致
2. **ADR の充実**: 4件の ADR が Context/Decision/Consequences の形式で記録されている。特に ADR-002 は DRY違反のトレードオフを明示しており、設計判断の透明性が高い
3. **状態遷移の二重防御**: disabled 属性 + 状態チェックの二重防御が設計されており、不正操作に対する堅牢性が考慮されている
4. **トレーサビリティの網羅**: コンポーネント表・状態表・シナリオすべてに traces フィールドがあり、FR/NFR との対応が追跡可能
5. **多角的な振る舞い記述**: シーケンス図（正常フロー・初期表示）+ アクティビティ図 + 状態遷移図の3種類で振る舞いを記述しており、設計の意図が多面的に伝わる
6. **Ch6 の設計原則確認観点**: 実装フェーズでの自己チェック項目が事前に定義されており、品質の作り込みに寄与する
