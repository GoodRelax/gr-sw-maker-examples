## 指摘一覧

### Medium 指摘

#### M-001: NFR-002 の測定基準が不明確

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1a (非機能要求の測定可能性) |
| 箇所 | NFR-002 (119行目) |
| 問題 | 「60fps を目標とし」という表現は目標であり必須基準ではない。また「滑らかに動作する」は主観的で測定不能。合格/不合格の判断基準が曖昧 |
| 影響 | テスト工程で合否判定が実施者依存になる |
| 修正案 | 「アニメーション中のフレームレートが 30fps を下回るフレームが全体の 5% 以下であること」のように、SHALL で定量基準を定義する。60fps は SHOULD（推奨）として併記可 |

#### M-002: FR-002 の演出仕様にタイミング基準がない

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b (完全性の検証 - 境界条件) |
| 箇所 | FR-002 (88-94行目) |
| 問題 | アニメーションの各段階（落下、転がり、停止、ズームイン）の所要時間や全体の持続時間が未定義。実装者の裁量幅が大きすぎる |
| 影響 | 実装ごとにアニメーション時間がばらつき、FR-005（再振りの有効化タイミング）やNFR-004（1秒以内の操作可能）との整合性が不確実になる |
| 修正案 | アニメーション全体の最大持続時間（例: 2秒以内）を定義する。または design フェーズで詳細化する旨の注釈を付記する |

#### M-003: 異常系・準正常系の考慮が不足

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b (異常系・準正常系の網羅) |
| 箇所 | 2.1 Functional Requirements 全体 |
| 問題 | 以下の準正常系シナリオに対する要求が未定義: (1) アニメーション中に「振る」ボタンが連打された場合の動作、(2) ページリサイズがアニメーション中に発生した場合の動作 |
| 影響 | 実装者が独自判断で対応し、テスト観点も漏れる可能性がある |
| 修正案 | FR-005 に「アニメーション実行中は『振る』ボタンを無効にする」旨を明示する。リサイズについては Limitations に記載するか、要求として定義する |

#### M-004: NFR-001 のレスポンシブ対応の検証基準が不十分

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1a (非機能要求の測定可能性) |
| 箇所 | NFR-001 (115-116行目) |
| 問題 | 「正常にレイアウトされること」の定義が曖昧。何をもって正常とするかの基準がない |
| 影響 | テスト時に合否判定が主観的になる |
| 修正案 | 「サイコロ2個と振るボタンが画面内に収まり、操作可能であること。要素の重なりやはみ出しが発生しないこと」のように具体化する |

### Low 指摘

#### L-001: FR-004 の色指定が曖昧

| 項目 | 内容 |
|------|------|
| 重大度 | Low |
| 観点 | R1b (曖昧性の排除) |
| 箇所 | FR-004 (101-102行目) |
| 問題 | 「赤色」「黒色」の具体的な色コードが未定義 |
| 影響 | 実装に支障はないが、design フェーズでの具体化が望ましい |
| 修正案 | design フェーズで色コードを指定する。または「麻雀サイコロとして一般的に認識される赤色」であればよい旨を記載する |

#### L-002: 用語集に「振るボタン」の定義がない

| 項目 | 内容 |
|------|------|
| 重大度 | Low |
| 観点 | R1a (用語の一貫性) |
| 箇所 | 1.8 Glossary |
| 問題 | FR-001 で「振る」ボタンと記載されているが、用語集には「振る (roll)」のみで UI 要素としてのボタンの定義がない |
| 影響 | 軽微。文脈から十分に理解可能 |
| 修正案 | 用語集に追加するか、Ch4 の UI Elements Map で定義する旨を注記する |

#### L-003: Constraints に否定形のみで代替動作の記載がない

| 項目 | 内容 |
|------|------|
| 重大度 | Low |
| 観点 | R1b (否定形の排除) |
| 箇所 | 1.6 Constraints (46-49行目) |
| 問題 | 「使用してはならない」「読み込んではならない」の否定形制約に対し、代わりに何をすべきか（静的ファイルのみ、インラインのみ）は記載されているが、3項目目の「単一HTMLファイルで完結すること」のみ肯定形 |
| 影響 | 軽微。否定形と肯定形が併記されており実質的に明確 |
| 修正案 | 記録のみ。現状で実用上問題なし |

#### L-004: NFR-003 に否定形の記述が残存

| 項目 | 内容 |
|------|------|
| 重大度 | Low |
| 観点 | R1b (否定形の排除) |
| 箇所 | NFR-003 (123-124行目) |
| 問題 | 「ネットワークリクエストを発行してはならない」の否定形に対する代替動作（ローカルリソースのみを使用する等）が明示されていない |
| 影響 | 軽微。Constraints 1.6 と合わせて読めば意図は明確 |
| 修正案 | 「すべてのリソースをHTMLファイル内に含めること」と肯定形で補足する |

#### L-005: 要求間の依存関係が暗黙的

| 項目 | 内容 |
|------|------|
| 重大度 | Low |
| 観点 | R1b (仕様の重複・競合の排除 - 依存関係の明示) |
| 箇所 | FR-005 と FR-002 の関係 |
| 問題 | FR-005 は「前回のアニメーションが完了する」をトリガーとしており、FR-002 のアニメーション仕様に暗黙的に依存しているが、明示的な参照（例: FR-002参照）がない |
| 影響 | 軽微。要求数が少ないため参照なしでも追跡可能 |
| 修正案 | FR-005 に「(FR-002 のアニメーション完了後)」と明示する |

## 良好な点

1. **テンプレート完全準拠**: ANMS v0.33 の Ch1 全9セクション、Ch2 全2セクションが過不足なく記述されている
2. **EARS構文の適切な使用**: 全要求が Event-driven または Ubiquitous パターンに正しく従っている
3. **Scope の明確な線引き**: In-scope / Out-of-scope が具体的に列挙されており、スコープクリープを防止できる
4. **Constraints と Limitations の区別**: 「破れない制約」と「許容する妥協」が適切に分離されている
5. **NFR の定量性**: NFR-004 (1秒以内)、NFR-005 (50KB以下) は明確な数値基準を持つ
6. **用語集の整備**: プロジェクト固有の5用語が英語対訳付きで定義されている
