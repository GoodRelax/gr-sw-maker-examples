## Findings (指摘一覧)

### High-001: 要求ID の採番が非連続で FR-05 以降が飛んでいる

| 項目 | 内容 |
|------|------|
| 重大度 | High |
| 観点 | R1a: 要求IDの一貫性 |
| 箇所 | Ch2.1 全体 |
| 問題 | 機能要求の掲載順序が FR-01, FR-06, FR-07, FR-08, FR-02, FR-03, FR-04, FR-05, FR-09 と非連続である。FR-06 ~ FR-09 が FR-01 の直後に配置され、FR-02 ~ FR-05 がその後に続く。ID の採番順と文書内の出現順が一致しておらず、トレーサビリティの確保が困難になる。レビューアや後続エージェント（architect, test-engineer）が要求を参照する際に混乱を招き、要求の見落としに直結する。 |
| 影響 | 要求の追跡性低下。設計・テストフェーズでの参照ミスのリスク。 |
| 修正案 | 以下のいずれかを実施する: (A) 機能要求を FR-01 ~ FR-09 の番号順に並べ替える、または (B) 機能的な関連性で論理グループ化し、各グループ内で番号順とする（例: 接続管理 FR-01/FR-09、スキャン FR-06/FR-07/FR-08、DTC FR-02/FR-03、データ FR-04/FR-05）。いずれの場合も採番順と記載順の対応ルールを明示すること。 |

### High-002: 非機能要求に測定不可能な表現が残っている

| 項目 | 内容 |
|------|------|
| 重大度 | High |
| 観点 | R1a: テスト不可能な表現の排除 / R1b: 曖昧性の排除 |
| 箇所 | NFR-02b (L265) |
| 問題 | 「アプリがクラッシュしても、それまでの記録データが失われないように」は、許容されるデータ損失量の定義がない。フラッシュ書き込みの頻度（毎行 / 毎秒 / バッファ単位）が未定義であり、テストの合否を判定できない。例えば「最大 1 秒分のデータ損失は許容する」のか「1 行も失わない」のかが不明。 |
| 影響 | テスト設計時に合否基準が定まらず、実装者とテスト担当で解釈が分かれる。 |
| 修正案 | 許容データ損失量を数値で定義する。例: 「クラッシュ時のデータ損失は最大 1 秒分（2 行分）以内とする。System shall 各書き込みの直後に fsync を実行するか、最大 1 秒間隔でバッファをフラッシュする。」 |

### Medium-001: FR-05b「500ms 間隔」と NFR-01b「1 秒以内」の関係が未整理

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b: 仕様の重複・競合の排除 |
| 箇所 | FR-05b (L216), NFR-01b (L259) |
| 問題 | FR-05b はデータ記録を 500ms 間隔と定義しているが、NFR-01b はダッシュボード更新を「1 秒以内の間隔（PID 数に依存）」と定義している。CON-02 で「全 PID を 500ms 以内に読み取れない場合がある」と制約されているため、PID 数が多い場合に 500ms 記録間隔を保証できない可能性がある。この場合の振る舞い（間隔延長 / 取得可能な PID のみ記録 / エラー通知）が未定義。 |
| 影響 | 実装判断がぶれ、テスト基準が曖昧になる。 |
| 修正案 | 500ms 間隔を「目標値」とし、全 PID 読取に 500ms 以上かかる場合の動作を明示する。例: 「全 PID の読取に 500ms 以上を要する場合、ラウンドロビン 1 周の完了をもって 1 記録行とし、実際の記録間隔をタイムスタンプに記録する。」 |

### Medium-002: EARS 構文の不統一（日英混在パターン）

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1a: EARS 構文の正しさ |
| 箇所 | Ch2.1 全体 |
| 問題 | EARS 構文のキーワード（When, While, If, then）は英語で記述されているが、主語 "the System" と動詞 "shall" の後の応答部分が日本語で記述されている。これ自体は許容されるが、一部の要求で構文パターンが不統一: (1) FR-01e は「If ... 場合, then the System shall ...」と「場合」を日本語で付加しているが、FR-02d では「If ... 場合, then the System shall ...」、FR-06f も同様。一方 FR-03c/FR-03d は「If ... 場合, then the System shall ...」。パターンは統一されているが、EARS の Unwanted Behavior パターンの正式構文は "If [Trigger], then the [System] shall [Response]." であり、日本語トリガー内に「場合」を含めるなら全箇所で統一すべき。 |
| 影響 | 軽微だが、パーサや自動トレーサビリティツールでの構文抽出に支障が出る可能性がある。 |
| 修正案 | EARS 構文パターンを統一する。推奨: 「If [日本語条件], then the System shall [日本語応答].」として「場合」は条件内に含める形に統一する。 |

### Medium-003: 境界条件が未定義（タイムアウト値、リトライ間隔）

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b: 完全性の検証 / 境界条件 |
| 箇所 | FR-01e (L147), NFR-02a (L264) |
| 問題 | FR-01e で「タイムアウト」と記載されているが、タイムアウトの具体的な秒数が未定義。NFR-02a で「3 回までリトライ」と記載されているが、リトライ間隔（即時 / 1 秒後 / 指数バックオフ等）が未定義。また、3 回リトライ後の動作が FR-01e のエラー表示に遷移するのかが要求間の依存として明示されていない。 |
| 影響 | 実装者がタイムアウト値とリトライ戦略を独自判断することになり、テスト条件が不明確になる。 |
| 修正案 | (1) タイムアウト値を数値で定義する（例: 「ELM327 応答タイムアウトは 5 秒とする」）。(2) リトライ間隔を定義する（例: 「リトライ間隔は 1 秒固定」または「指数バックオフ（1s, 2s, 4s）」）。(3) NFR-02a に「3 回リトライ後も失敗した場合は FR-01e のエラー処理に従う」と依存関係を明示する。 |

### Medium-004: リソース枯渇時の振る舞いが未定義

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b: 異常系・準正常系の網羅 |
| 箇所 | FR-05 全体 |
| 問題 | データ記録（TSV ファイル書き込み）において、ディスク容量不足になった場合の振る舞いが定義されていない。長時間の記録ではファイルサイズが増大するが、ディスクフル時にアプリがどう振る舞うか（エラー通知して記録停止 / 古いデータを上書き / 等）が不明。 |
| 影響 | ディスクフル時にデータ損失やアプリクラッシュが発生する可能性がある。 |
| 修正案 | FR-05 に異常系要求を追加する。例: 「If TSV ファイルの書き込みに失敗した場合（ディスク容量不足等）, then the System shall 記録を停止し、エラーメッセージを表示し、それまでの記録データを保全する。」 |

### Medium-005: キャッシュファイルの最大サイズ・有効期限が未定義

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b: 完全性の検証 / 境界条件 |
| 箇所 | FR-07a (L161) |
| 問題 | スキャン結果キャッシュ（`~/.car-diag/cache/` 配下の JSON ファイル）の有効期限や最大サイズが未定義。複数車両のキャッシュが蓄積された場合のディスク使用量の上限や、古いキャッシュの自動削除ポリシーが不明。また、車両の ECU 構成が変更された場合（ECU 交換等）に古いキャッシュが誤動作を招く可能性がある。 |
| 影響 | 長期使用時のディスク消費増大、および古いキャッシュによる誤動作の可能性。 |
| 修正案 | キャッシュの有効期限（例: 「30 日間アクセスがないキャッシュは自動削除する」）、または最大エントリ数を定義する。キャッシュ無効化のトリガー（例: 「ECU スキャン結果が前回と異なる場合はキャッシュを更新する」）も明示する。 |

### Medium-006: DID 全スキャンの所要時間見積もりと中断の安全性が不十分

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b: 完全性の検証 |
| 箇所 | FR-08c (L170) |
| 問題 | DID 全スキャン（0x0000 ~ 0xFFFF = 65,536 DID x ECU 数）の所要時間の見積もりが制約事項に記載されていない。CON-03 に KWP2000 の速度制約はあるが、UDS の DID 全スキャンにかかる時間が制約として明示されていない。ユーザーが所要時間を認識せずに全スキャンを開始する可能性がある。 |
| 影響 | ユーザー体験の低下。想定外の長時間実行によるユーザー混乱。 |
| 修正案 | (1) CON セクションに DID 全スキャンの所要時間の見積もりを追加する（例: 「DID 全スキャンは 1 ECU あたり数時間を要する場合がある」）。(2) FR-08c に「全スキャン開始前に推定所要時間を表示し、ユーザーに確認を求める」を追加する。 |

## Positive Observations (良い点)

1. **EARS 構文の適用が徹底されている。** 全機能要求が When / While / If / Ubiquitous のいずれかのパターンで記述されており、可読性と機械処理性が高い。
2. **マルチプロトコル対応の仕様が詳細かつ具体的。** Legacy OBD / UDS / KWP2000 それぞれの SID とコマンドが明確に定義されており、実装の曖昧さが少ない。
3. **接続状態管理（FR-09）で FSA が明示的に定義されている。** 状態と遷移の網羅性が高く、特に全状態からの通信断ハンドリング（FR-09a）が包括的に記述されている。
4. **Constraints と Limitations の区別が適切。** CON（破れない制約）と LIM（許容する妥協点）が明確に分離されている。
5. **用語集が充実しており、OBD-II ドメインの主要用語が網羅されている。**

High 指摘 2 件が存在するため、planning フェーズから design フェーズへの遷移をブロックする。

**推奨戻り先:** srs-writer エージェントによる仕様書 Ch1-2 修正（planning フェーズ相当）。
