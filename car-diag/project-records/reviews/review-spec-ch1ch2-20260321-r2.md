## 新規指摘 (Findings)

### Medium-007: FR-02a の参照先が旧番号体系の可能性（FR-05e / FR-09 参照の正確性）

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1a: 要求IDの一貫性 / R1b: 仕様の重複・競合の排除 |
| 箇所 | FR-02a (L179-180) |
| 問題 | FR-02a 内で「DID スキャン中の場合は中断位置をキャッシュに保存する（FR-05e）」と参照しているが、FR-05e（L217）は「ユーザーが DID スキャンを中断した時」のキャッシュ保存を定義しており、通信断による自動中断のケースを明示的にカバーしているかが曖昧である。FR-05e のトリガーが「ユーザーが中断」であるのに対し、FR-02a のトリガーは「通信断」であり、厳密にはトリガー条件が異なる。また「FR-09 データ損失防止」と記載されているが、データ損失防止の具体的な定義は FR-09f にある。参照先を FR-09f と明記すべき。 |
| 影響 | 実装者が通信断時の DID スキャンキャッシュ保存を実装し忘れる可能性がある。 |
| 修正案 | (1) FR-05e のトリガーを「ユーザーが DID スキャンを中断した時、または通信断により DID スキャンが中断された時」に拡張する。(2) FR-02a の「FR-09 データ損失防止」を「FR-09f のフラッシュ間隔に従う」に修正する。 |

### Medium-008: FR-04d のキャッシュキー衝突リスクが未考慮

| 項目 | 内容 |
|------|------|
| 重大度 | Medium |
| 観点 | R1b: 異常系・準正常系の網羅 |
| 箇所 | FR-04d (L202) |
| 問題 | VIN が取得できない車両に対してユーザーが任意の識別名を入力しキャッシュキーとする仕様だが、ユーザーが異なる車両に同一の識別名を入力した場合のキャッシュ衝突が考慮されていない。また、識別名に使用可能な文字種やファイル名として有効な制約（パス区切り文字やスペース等の除外）が未定義。 |
| 影響 | キャッシュの誤適用（別車両のスキャン結果が読み込まれる）や、ファイル名として不正な文字列によるファイル書込み失敗の可能性がある。 |
| 修正案 | (1) 識別名の文字種制約を定義する（例: 「半角英数字・ハイフン・アンダースコアのみ、最大 32 文字」）。(2) 同一識別名のキャッシュが既に存在する場合の動作を定義する（例: 「上書き確認ダイアログを表示する」）。 |

## Positive Observations (良い点)

前回レビューの良い点に加え、以下の改善が認められる:

1. **FR の論理的順序化が適切。** 接続→状態管理→スキャン→キャッシュ→DID→DTC→ダッシュボード→記録という操作フロー順の配置が明確で、要求の追跡性が向上した。
2. **NFR-02b の数値化が的確。** 「最大 1 秒分（2 行分）」という定量的基準と FR-09f への参照により、テスト可能な要求になっている。
3. **異常系要求の充実。** FR-09f（fsync 間隔）、FR-09g（ディスクフル）、CON-09（DID スキャン所要時間）、FR-04e/FR-04f（キャッシュ管理）が追加され、要求の完全性が向上した。
4. **要求間の依存関係が明示化。** NFR-02a→FR-01e、FR-05a→FR-03、FR-05c→CON-09 など、クロスリファレンスが適切に記載されている。
5. **用語集と参考文献の網羅性が向上。** VIN・NRC の追加、ISO 14229 の追加により、ドメイン知識のないレビューアにも理解可能な文書になった。
